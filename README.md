# Ledger — SPA «Управління проектами і завданнями»

Тестове завдання для Frontend-розробника (Vue.js 3 / TypeScript). SPA для
управління проектами та завданнями всередині них, без реального бекенду —
дані живуть у `localStorage` за мок-адаптером Axios.

## 🔗 Розгорнута версія

> Додайте посилання після деплою, наприклад:
> `https://ledger-tasks.vercel.app`

## 🚀 Локальний запуск

```bash
npm install
npm run dev
```

Застосунок підніметься на `http://localhost:5173`. Дані одразу заповнюються
демо-проектами та завданнями (перший запуск сідить `localStorage`
автоматично, дивись `src/api/localDb.ts`).

Інші команди:

```bash
npm run build      # production-збірка (vue-tsc + vite build)
npm run preview    # локальний перегляд production-збірки
npm run typecheck  # перевірка типів без збірки
```

## 🧱 Технологічний стек

| Категорія   | Технологія                          |
| ----------- | ------------------------------------ |
| Фреймворк   | Vue 3 (Composition API, `<script setup>`) |
| Мова        | TypeScript, строгий режим, без `any` |
| Стан        | Pinia (`setup`-стори)                |
| HTTP        | Axios з кастомним mock-адаптером     |
| Стилі       | SCSS (дизайн-токени + BEM-подібні класи) |
| Drag & Drop | `vuedraggable` (SortableJS)          |
| Валідація   | `vee-validate` + `zod` (`@vee-validate/zod`) |
| Дані        | `localStorage` + мок-адаптер з затримкою 150–300мс |

Жодного `any` у коді немає — усі типи прописані в `src/types/`, generic-и
використовуються в API-шарі (`api.get<T>`, `api.post<T, B>` тощо).

## 🏗️ Архітектурні рішення

### Шар API (`src/api/`)

- **`http.ts`** — тонка обгортка над Axios-інстансом з generic-методами
  (`get<T>`, `post<T, B>`, `put<T, B>`, `delete<T>`). Усі axios/мок-помилки
  тут-таки нормалізуються в `AppError { message, status }`, тож компоненти й
  стори ніколи не бачать `AxiosError` — обробка помилок повністю на рівні
  сервісного шару, як і вимагалося в завданні.
- **`mockAdapter.ts`** — кастомний Axios `adapter`, який підміняє мережевий
  рівень: парсить `method + url`, симулює затримку 150–300мс і читає/пише
  дані через `localDb.ts`. Реалізовані всі обов'язкові ендпоінти
  (`GET/POST/PUT/DELETE /projects`, `GET/POST/PUT/DELETE /tasks`,
  включно з фільтром `?projectId=`). Якщо колись з'явиться реальний
  бекенд — достатньо прибрати `adapter: mockAdapter` у `http.ts`, і решта
  застосунку не зміниться, бо компоненти працюють лише з `projectsApi` /
  `tasksApi`.
- **`localDb.ts`** — низькорівневий доступ до `localStorage` (читання,
  запис, генерація ID, сідування демо-даних один раз при першому запуску).
- **`projectsApi.ts` / `tasksApi.ts`** — тонкі, повністю типізовані фасади
  над `api`, це єдине місце, звідки компоненти й стори звертаються до
  «мережі».

### Стан (`src/stores/`)

- **`projects.ts`** і **`tasks.ts`** — окремі Pinia-стори (Composition API
  синтаксис `defineStore(id, () => {...})`).
- `tasks.ts` тримає єдиний масив `tasks` для поточного проекту й похідний
  `computed` `byStatus`, згрупований по статусу. Таблиця і канбан
  **використовують один і той самий стор**, тож зміна статусу в канбані
  миттєво відображається в таблиці й навпаки — без ручної синхронізації.
- **`toast.ts`** — окремий стор для тост-повідомлень (бонусна фіча),
  навмисно відокремлений від бізнес-даних.

### Composables (`src/composables/`)

- **`useTableSort.ts`** — generic сортування «клік по заголовку» для будь-
  якої таблиці; приймає `accessor(item, field)`, тож однаково працює і для
  проектів, і для завдань.
- **`useColumnResize.ts`** — drag-to-resize для заголовків колонок через
  Pointer Events, повертає реактивну мапу `widths`.
- **`useTaskDrag.ts`** — місток між подіями `vuedraggable` (kanban-колонки
  та рядки таблиці) і Pinia-стором завдань: саме тут вирішується, як
  перетягування картки перетворюється на `moveTask` / `reorderWithinStatus`.
- **`usePersistedState.ts`** — generic `ref`, що синхронізується з
  `localStorage` (бонус: фільтри, сортування й обраний режим перегляду
  зберігаються між перезавантаженнями).

### Компоненти (`src/components/`)

Розбиті за відповідальністю, без God-компонентів:

```
components/
├── common/       BaseModal, ToastContainer, StatsChart (перевикористовувані, без бізнес-логіки)
├── projects/      ProjectsTable, ProjectFormModal, ProjectStatusBadge
└── tasks/         TasksTable, TaskFormModal, KanbanBoard, KanbanColumn, TaskCard
```

`views/ProjectsView.vue` і `views/ProjectDetailView.vue` — «розумні»
сторінки, які знають про стори й роутинг; самі компоненти в
`components/*` — «дурні», отримують дані через props і повідомляють про дії
через emit. Модальні форми (`ProjectFormModal`, `TaskFormModal`)
самодостатні: інкапсулюють `vee-validate` + `zod`-схему й емітять уже
провалідований payload.

### Типізація (`src/types/`)

- `Project`, `Task` — інтерфейси з усіма полями.
- `ProjectStatus`, `TaskStatus` — enum-и.
- `ProjectSortField`, `TaskSortField`, `SortState<T>` — допоміжні generic-
  типи для composables.

## 🖱️ Drag & Drop

- **Канбан**: `vuedraggable` з `group="tasks"` дозволяє тягати картки як у
  межах колонки (зміна порядку), так і між колонками (зміна статусу +
  позиції). Подія `change` веде до `useTaskDrag().onKanbanChange`, який
  викликає `tasksStore.moveTask`.
- **Таблиця завдань**: рядки тягаються за «ручку» (⠿), що викликає
  `tasksStore.reorderWithinStatus` для кожного статусу, присутнього у
  видимому списку. Перетягування вимкнене, поки активне сортування за
  колонкою (інакше нема сенсу — порядок все одно перевизначиться сортом);
  про це є підказка в інтерфейсі.
- **Колонки** обох таблиць (проектів і завдань) розтягуються перетягуванням
  правого краю заголовка (`useColumnResize`).

## ✅ Валідація форм

`vee-validate` + `zod`-схеми (`src/utils/validation.ts`), помилки
відображаються під полями як при `blur`, так і при спробі відправки
(поведінка `vee-validate` за замовчуванням — `validateOnBlur` + перевірка на
`submit`). Правила відповідають ТЗ: назва проекту 2–100 символів, назва
завдання 3–120, термін виконання не раніше сьогодні, статус — обов'язковий
enum.

## 🎁 Реалізований бонусний функціонал

- Toast-повідомлення про дії (додавання/видалення/оновлення).
- Діаграма розподілу завдань за статусами на головній сторінці.
- Фільтри, сортування й обраний режим перегляду (таблиця/канбан)
  зберігаються між перезавантаженнями через `localStorage`.
- Легкі transition-анімації для рядків таблиці, тостів і модальних вікон.

## ⚠️ Свідомі спрощення

- Список виконавців — фіксований масив `TEAM_MEMBERS`
  (`src/utils/constants.ts`) замість окремого API довідника — для тестового
  завдання без бекенду це виправдано.
- Перетягування рядків таблиці синхронізує `order` лише в межах статусів,
  присутніх серед відфільтрованих рядків: це свідомий компроміс, щоб не
  переносити приховані фільтром рядки випадково.
- `any` у коді відсутній повністю, обґрунтування не знадобилось.
